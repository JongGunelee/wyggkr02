- 작업 루트: `D:\03 금일작업\00 임시\00000 스크립트\깃허브`

## 1. 작업 배경

기존 구조는 `000 Launch_dashboard.bat`와 `run_dashboard.py`를 통해 로컬에서 `00 dashboard.html`을 same-origin 방식으로 여는 구조였습니다.

사용자 요구는 다음이었습니다.

1. `system_guides/00 PRD 가이드.md`를 우선 정독
2. 관련 가이드 및 `000 implementation_plan_웹02.md` 분석
3. `../01 Scripts` 원본 구조와 `000 Launch_dashboard.bat -> 00 dashboard.html` 구현 방식 분석
4. 현재 `깃허브` 복사본의 `00 dashboard.html`을 웹용으로 전환
5. 무결성 보증 리팩토링 수행

이후 추가 요구로 아래가 이어졌습니다.

1. BAT 없이 동작 가능한 설치형 EXE 구조로 개편
2. 웹 배포 + 로컬 실행 에이전트 구조 확립
3. GitHub 업로드/Release 배포 방식 정리
4. 카드 실행 오류(`tkinter`) 대응
5. EXE 재실행 시 대시보드 재활성화/혼선 제거
6. 폴더 정리 및 개발 보관 파일 분리

## 2. 초기 분석 결과

- 이번 작업은 원본을 건드리지 않고 `깃허브` 복사본에서만 수행

### 2.2 기존 동작 방식

- `00 dashboard.html`은 원래 로컬 서버 same-origin 전제

### 2.3 구현 계획서 해석

`system_guides/000 implementation_plan_웹02.md` 기준 목표 아키텍처는:

- UI: GitHub Pages
- 실행 주체: 사용자 PC 로컬 에이전트(`127.0.0.1:8501`)
- 카드 클릭 시:
  - 웹에서는 자산 열람
  - 로컬에서는 실제 스크립트 실행

## 4. 2차 작업: BAT 의존 제거 및 설치형 EXE화

### 4.1 문제

브라우저는 웹페이지에서 로컬 프로그램을 직접 실행할 수 없으므로, BAT 없이도 결국 로컬 런처는 필요함.

BAT 대신 사용자가 더 쉽게 실행할 수 있는 설치형 EXE로 전환함.

### 4.2 추가/변경 파일

- `dev_source/dashboard_agent_launcher.py`
- `dev_source/dashboard_agent_launcher.spec`
- `dev_source/build_dashboard_agent.ps1`
- `dev_source/run_dashboard.py`

### 4.3 EXE 런처 구조

`dashboard_agent_launcher.py`의 역할:

- 일반 실행 시:
  - 로컬 에이전트 기동
- `--run-script <상대경로>` 시:
  - 내장 스크립트 직접 실행

### 4.4 EXE 패키징 구조

### 4.5 빌드 표준화

`build_dashboard_agent.ps1`에서:

  - `WYGGKR02_Dashboard_Agent_Setup.exe`

### 4.6 런타임 구조 개선

## 6. 4차 작업: `tkinter` 누락 오류 수정

### 6.1 증상

웹 대시보드의 카드 섹션에서 `로컬 실행` 클릭 시 EXE 내부 스크립트가 다음 오류로 종료:

- `No module named 'tkinter'`

### 6.2 원인

다수 자동화 스크립트가 `tkinter`, `ttk`, `filedialog`, `messagebox`를 사용하지만 EXE spec에 명시적으로 포함되지 않았음.

### 6.3 조치

`dashboard_agent_launcher.spec`에 다음 추가:

- `tkinter`
- `tkinter.ttk`
- `tkinter.filedialog`
- `tkinter.messagebox`
- `tkinter.simpledialog`
- `_tkinter`

### 6.4 결과

`tkinter` 기반 카드가 EXE 내부에서 정상 기동됨

## 7. 5차 작업: GitHub 업로드 제한 대응

### 7.1 문제

`WYGGKR02_Dashboard_Agent_Setup.exe`는 저장소/Pages에 직접 올리기 어려움.

### 7.2 대응 방식

- 웹 파일은 GitHub Pages
- 설치 파일은 GitHub Release 자산

### 7.3 링크 변경

`00 dashboard.html`의 다운로드 링크를 저장소 상대경로에서 Release 다운로드 URL로 변경:

- `https://github.com/jonggunelee/wyggkr02/releases/latest/download/WYGGKR02_Dashboard_Agent_Setup.zip`

### 7.4 ZIP 배포 전환 이유

브라우저가 `.exe`를 직접 취급할 때 `사이트에서 사용할 수 없는 파일` 경고를 보여줄 수 있음.

따라서 릴리스 자산은:

- `WYGGKR02_Dashboard_Agent_Setup.zip`

형태로 제공하도록 변경

### 7.5 현재 사용자 동선

1. 웹에서 ZIP 다운로드
2. 압축 해제
3. 내부 `WYGGKR02_Dashboard_Agent_Setup.exe` 실행
4. 웹 탭에서 `연결 재확인`

## 8. 6차 작업: EXE 재실행 시 대시보드 재활성화 혼선 제거

### 8.1 문제

사용자가 이미 웹 대시보드를 열어 둔 상태에서 EXE를 다시 실행하면:

- 기존 웹 탭이 다시 활성화되거나
- 새 브라우저 탭/포커스 개입이 발생해 혼선 유발

### 8.2 조치

`dashboard_agent_launcher.py`에서:

- `webbrowser.open()` 기반 자동 웹 오픈 제거
- 대기 스레드 기반 브라우저 자동 실행 제거
- 이미 에이전트가 동작 중이면 새 브라우저를 열지 않고 복귀
- 전역 뮤텍스 도입:
  - `Global\\WYGGKR02_Dashboard_Agent_Launcher`

### 8.3 결과

EXE는 이제 브라우저를 다시 열지 않고 로컬 에이전트만 기동함.

즉:

- 웹 탭은 사용자가 직접 열어 둔 것을 그대로 사용
- EXE는 백엔드 역할만 수행

## 11. 현재 최종 상태 요약

현재 구조에서 가능한 것:

- 웹에서 대시보드 열기
- ZIP 다운로드
- 압축 해제 후 EXE 실행
- Python 미설치 환경에서 카드 `로컬 실행`
- `tkinter` GUI 기반 카드 실행
- 브라우저 자동 재오픈 없이 로컬 에이전트만 기동

현재 필요한 로컬 조건:

- Windows 환경
- Office 관련 카드 사용 시 MS Office 설치
- EXE 실행 허용
- 로컬 보안 정책이 `127.0.0.1:8501` 통신을 차단하지 않을 것

현재 불필요한 것:

- 로컬 Python 설치
- pip로 라이브러리 수동 설치
- BAT 실행

## 13. 운영 시 주의사항

1. 웹 업로드와 Release 업로드는 분리해야 함
2. Release에는 가능하면 EXE가 아니라 ZIP을 올릴 것
3. 파일명은 유지해야 함
   - `WYGGKR02_Dashboard_Agent_Setup.zip`
   - 내부 EXE: `WYGGKR02_Dashboard_Agent_Setup.exe`
4. 새 EXE를 배포할 때는 ZIP도 같이 다시 갱신할 것
5. 사용자는 기존 ZIP/EXE를 지우고 최신 버전으로 다시 받아야 함

- 작업 범위: `깃허브` 복사본 기준 웹 전환, 로컬 에이전트 EXE 패키징, 배포 동선 정리, 중복 실행/다운로드 이슈 대응
- 작업 루트: `D:\03 금일작업\00 임시\00000 스크립트\깃허브`

## 0. 최우선 운영 규칙 (처음 읽기)

1. 루트(`깃허브/`)는 웹 배포 파일 전용으로 유지한다.
2. 루트에 유지되는 파일은 `00 dashboard.html`, `index.html`, `manifest.webmanifest`, `service-worker.js`로 제한한다.
3. 빌드 산출물(EXE/ZIP), 원격 동기화 캐시, 임시 파일은 모두 `dev_source/__temp_*` 하위에서만 생성/관리한다.
4. GitHub 저장소 업로드 대상은 현재 운영 루트 전체 기준(`automated_app/`, `automated_scripts/`, `runtime_store/`, `system_guides/`, `dev_source/` 핵심 파일 + 웹 루트 파일)으로 관리한다.
5. `dev_source/runtime_store/`에는 GitHub 개별 파일 업로드 제한(25MB 이하)에 맞는 wheel과 매니페스트만 둔다.
6. 25MB 초과 자산은 `runtime_manifest.json`의 `parts` 방식으로 분할하거나, 공식 배포처 URL을 통해 최초 1회 내려받도록 설계한다.
7. 임시 정리는 `dev_source/cleanup_temp_workdirs.ps1`를 사용하고, 빌드 파이프라인 시작/종료 시 자동 실행한다.
8. 임시 파일 정리 시 루트/업무 파일을 패턴 삭제하지 않는다. 삭제 범위는 `dev_source/__temp_artifacts__/WYGGKR02_Dashboard_Agent_Setup.__tmp__*` 및 `dev_source/__temp_artifacts__/WYGGKR02_Dashboard_Agent_Setup.__tmp_stage__*`로 고정한다.
9. `manifest.webmanifest`, `service-worker.js`는 웹 경로 참조상 루트 필수 파일이므로 이동 금지한다.
10. 배포는 "무조건 전체 업로드"가 아니라 "전체 폴더 스캔 + 변경분만 업로드"를 기본 정책으로 한다.
11. `publish_repo_assets.ps1`는 로컬 파일의 Git blob SHA(바이트 기준)를 계산해 원격 SHA와 같으면 업로드를 건너뛴다.
12. 따라서 동일 파일 재배포 시 커밋/API 호출/업로드 시간을 줄이고, 변경·추가·수정 파일만 반영된다.

### 0.1 현재 확정된 목적

이 프로젝트의 현재 목적은 아래 3가지를 동시에 만족하는 것이다.

1. 웹 UI는 GitHub Pages에서 항상 동일하게 열린다.
2. 실제 실행은 사용자 PC의 로컬 에이전트가 담당한다.
3. 변경 빈도가 높은 스크립트와 런타임 의존성은 GitHub 저장소에서 직접 갱신하고, Release는 최소 부트스트랩 EXE만 유지한다.

### 0.2 향후 코딩 AI가 절대 헷갈리면 안 되는 기준

1. `automated_scripts/` 수정은 원칙적으로 EXE 재빌드 사유가 아니다.
2. `dev_source/runtime_store/` 수정도 원칙적으로 EXE 재빌드 사유가 아니다.
3. EXE 재빌드는 `dashboard_agent_launcher.py`, `dashboard_agent_launcher.spec`, `run_dashboard.py`, 웹 루트 자산처럼 코어 부트스트랩/서버 동작이 바뀔 때만 한다.
4. GitHub 저장소와 GitHub Release는 역할이 다르다.
5. 저장소는 원격 소스 오브 트루스이고, Release는 사용자 다운로드 패키지다.

### 0.3 이번 즉시 진행 계획 (2026-04-09 추가)

목표:

1. 현재 경량 구조는 유지한다.
2. 다만 `automated_scripts/`를 EXE에 fallback 번들로 추가한다.
3. 런타임 원격 동기화는 계속 유지한다.

적용 순서:

1. 문서에 계획 선반영
2. `dashboard_agent_launcher.spec`에 `automated_scripts` 번들 항목 추가
3. EXE 재빌드/ZIP 재패키징
4. GitHub 저장소 + Release 동시 반영
5. 문서에 실제 결과(용량/해시/장단점) 확정 반영

기대 효과:

1. 최초 카드 실행 체감 속도 향상
2. 원격 동기화 실패 시 기본 스크립트 fallback 확보
3. 오프라인 초기 대응성 개선(기본 번들 스크립트 한정)

### 0.4 스마트 전체 배포 정책 (2026-04-09 확정)

정의:

1. 배포 기준 범위는 "전체 운영 폴더(루트 파일 + 주요 폴더)"다.
2. 업로드 방식은 "증분"이다. 동일 파일은 건너뛴다.

판정 방식:

1. 로컬 파일 바이트로 Git blob SHA(`blob <size>\0<bytes>` SHA1)를 계산한다.
2. GitHub Contents API의 원격 `sha`, `size`와 비교한다.
3. `sha`와 `size`가 모두 같으면 `[SKIP] Unchanged`로 건너뛴다.
4. 다르면 업로드하고 `[OK] Published`로 반영한다.

운영 효과:

1. 전체 폴더를 매번 검사해 누락 위험은 줄인다.
2. 변경 없는 파일은 재업로드하지 않아 속도/안정성이 올라간다.
3. 대형 wheel이 많은 `automated_app/` 구간에서 재배포 시간을 크게 줄일 수 있다.

## 6. 4차 작업: `tkinter` 누락 오류 수정

### 6.2 원인

### 6.3 조치

- `tkinter`

- `tkinter.filedialog`
- `tkinter.messagebox`
- `tkinter.simpledialog`
- `_tkinter`

### 6.4 결과

## 8. 6차 작업: EXE 재실행 시 대시보드 재활성화 혼선 제거

### 8.1 문제

사용자가 이미 웹 대시보드를 열어 둔 상태에서 EXE를 다시 실행하면:

- 기존 웹 탭이 다시 활성화되거나
- 새 브라우저 탭/포커스 개입이 발생해 혼선 유발

### 8.2 조치

`dashboard_agent_launcher.py`에서:

- `webbrowser.open()` 기반 자동 웹 오픈 제거
- 대기 스레드 기반 브라우저 자동 실행 제거
- 이미 에이전트가 동작 중이면 새 브라우저를 열지 않고 복귀
- 전역 뮤텍스 도입:
  - `Global\\WYGGKR02_Dashboard_Agent_Launcher`

### 8.3 결과

EXE는 이제 브라우저를 다시 열지 않고 로컬 에이전트만 기동함.

즉:

- 웹 탭은 사용자가 직접 열어 둔 것을 그대로 사용
- EXE는 백엔드 역할만 수행

교훈:

### 23.2 GitHub API 호출 401/403/409 오류

#### 23.2.1 `401 Unauthorized`

원인:

- `GITHUB_TOKEN` 미설정 또는 만료/폐기된 토큰 사용

대응:

- `GITHUB_TOKEN`을 다시 설정한 뒤 재시도한다.

#### 23.2.2 `403 Resource not accessible by personal access token`

원인:

- Fine-grained PAT 권한 부족(Contents write 미부여 등) 또는 대상 저장소 접근 권한 부재

대응:

- Fine-grained PAT 권한을 아래로 맞춘다:
  - Repository: 대상 repo 선택
  - Permissions: `Contents: Read and write`, `Metadata: Read`

#### 23.2.3 `409 ... expected <sha>`

원인:

- GitHub Contents API는 업데이트 시 원격 `sha`가 필요하며,
  직전 업로드와 경합(동일 파일을 연속/병렬로 갱신)되면 SHA mismatch가 발생함

해결:

- `publish_repo_assets.ps1`에 409 재시도 로직을 추가한다:
  - 원격 메타데이터(SHA) 재조회 후 재업로드

교훈:

- 배포는 단일 파이프라인(`build_and_release.ps1`) 한 경로로만 수행하고,
  중복 실행(이중 타격)을 피한다.

### 23.3 GitHub API 연결 거부(127.0.0.1:9) / SSL 오류

증상:

- GitHub API 호출 시 `127.0.0.1:9`로 연결 시도 후 거부
- 또는 SSL handshake 실패

원인:

- 환경변수 프록시가 잘못 설정되어(`HTTP_PROXY`, `HTTPS_PROXY`, `ALL_PROXY` 등)
  GitHub 요청이 로컬 프록시로 리다이렉트됨

대응:

- 배포 전 세션에서 아래 프록시 환경변수를 제거 후 재시도:

```powershell
$names='ALL_PROXY','GIT_HTTP_PROXY','GIT_HTTPS_PROXY','HTTP_PROXY','HTTPS_PROXY'
foreach($n in $names){ Remove-Item Env:$n -ErrorAction SilentlyContinue }
```

교훈:

- 배포 스크립트 실행 전 "프록시 환경변수 점검"을 습관화한다.

## 28. GitHub 한글 경로 깨짐 문제 원인/해결/교육

### 28.1 증상

GitHub 웹 저장소의 `dev_source/` 아래에서 일부 한글 파일/폴더명이 아래처럼 깨져 보이는 문제가 발생했다.

- `웹접속 주소.txt` → `?뱀젒??二쇱냼.txt`
- `__01 진행현황__` → `__01 吏꾪뻾?꾪솴__`

### 28.2 실제 원인

문제는 로컬 파일 자체가 깨진 것이 아니라, `publish_repo_assets.ps1`가 GitHub Contents API로 업로드할 때 원격 경로(`RepoPath`)를 한글 문자열로 직접 하드코딩한 데서 발생했다.

즉:

1. 로컬 파일명/폴더명은 정상 Unicode였다.
2. 그러나 자동 업로드 스크립트 내부의 한글 리터럴이 PowerShell/REST 업로드 경계에서 mojibake 상태로 해석된 케이스가 발생했다.
3. 그 결과 GitHub에는 “정상 한글 경로”가 아니라 “깨진 별칭 경로”가 새 파일/폴더로 생성되었다.

핵심:

- 파일 내용 인코딩 문제라기보다 “원격 경로 문자열 인코딩 문제”였다.
- 콘솔 출력이 깨진 것과 실제 GitHub 경로가 깨진 것은 구분해야 한다.
- 이번 건은 실제 GitHub 경로까지 잘못 생성된 케이스였다.

### 28.3 이번에 적용한 해결

`publish_repo_assets.ps1`를 아래 방식으로 수정했다.

1. 원격 경로의 한글 이름을 스크립트에 직접 하드코딩하지 않는다.
2. `Get-ChildItem`으로 실제 로컬 파일/폴더를 찾는다.
3. 찾은 객체의 `.Name` 값을 그대로 사용해 원격 `RepoPath`를 조합한다.

적용 예:

- `dev_source/웹접속 주소.txt`
  - 이전: 문자열 하드코딩
  - 현재: `$webUrlFile.Name` 사용
- `dev_source/__01 진행현황__/04 ... .md`
  - 이전: 폴더명/파일명 하드코딩
  - 현재: `$handoffDir.Name`, `$handoffFile.Name` 사용

그 후 GitHub API로 잘못 올라간 원격 경로를 삭제했다.

### 28.4 삭제/정정한 원격 경로

삭제:

- `dev_source/?뱀젒??二쇱냼.txt`
- `dev_source/__01 吏꾪뻾?꾪솴__/04 인수인계서_대시보드_웹전환_설치형EXE_2026-04-08.md`

유지:

- `dev_source/웹접속 주소.txt`
- `dev_source/__01 진행현황__/04 인수인계서_대시보드_웹전환_설치형EXE_2026-04-08.md`

### 28.5 향후 코딩 AI 교육 규칙

1. GitHub Contents API 업로드 시 한글 경로를 문자열로 직접 하드코딩하지 않는다.
2. 가능하면 로컬 파일 시스템에서 찾은 실제 `.Name`을 사용해 원격 경로를 조합한다.
3. 업로드 직후 반드시 원격 `contents` API로 해당 폴더 목록을 다시 조회한다.
4. 한글 경로가 포함된 자동 업로드에서는 “콘솔 출력만 깨진 것인지”, “실제 GitHub 경로가 깨진 것인지”를 분리 검증한다.
5. 경로 안정성이 최우선이면 장기적으로는 자동 업로드용 원격 파일명은 ASCII 별칭 정책으로 전환하는 것도 검토할 수 있다.

### 28.6 재발 방지 체크리스트

1. `publish_repo_assets.ps1` 수정 시 한글 `RepoPath` 하드코딩이 다시 들어가지 않았는지 검사
2. 업로드 후 `dev_source` 루트 목록에서 아래 2개가 정상 표기되는지 확인
   - `웹접속 주소.txt`
   - `__01 진행현황__`
3. 깨진 별칭 경로가 새로 생기면 즉시 삭제 후 스크립트 경로 조합 로직부터 점검

🛡️ 통합 병합 프리징 해결 및 UI 쓰레드 안정성 확보 가이드 [v35.4.18]

대량의 파일을 처리하거나 복잡한 경로 환경에서 발생하는 애플리케이션 무응답(Hang) 현상을 원천적으로 해결한 UI 안정성 하드닝 가이드입니다.

1. UI 쓰레드 안정성 확보 (Thread-Safe Callback)

현상: 작업 스레드(Background thread)에서 직접 UI 위젯(로그창, 진행바 등)을 조작할 때, 메인 스레드와의 경합으로 인해 윈도우가 '먹통'이 되는 교착 상태(Deadlock)가 발생했습니다.
해결: root.after(0, callback) 패턴을 전면 도입하여, 모든 UI 업데이트 명령을 메인 루프로 안전하게 전달함으로써 장시간 연산 중에도 UI가 멈추지 않고 실시간 응답성을 유지하도록 개선했습니다.

2. 오피스 대화상자 개입 원천 차단 (Interactive Mode Disabled)

현상: 파일을 열 때 "연결 업데이트", "보안 경고" 등 보이지 않는 팝업이 백그라운드에서 발생하여 프로세스가 무한 대기하는 현상이 발견되었습니다.
해결: COM 엔진 가동 시 app.Interactive = False를 강제 주입하여, 모든 경고나 알림창을 무시하고 자동 진행되도록 차단함으로써 무인 자동화의 연속성을 확보했습니다.

3. 경로 호환성 정규화 (Safe Path Normalization)

현상: 롱패스 지원용 \\?\ 접두사가 260자 미만의 짧은 경로에 붙어 있을 때, 일부 Office 버전에서 파일을 열지 못하는 호환성 문제가 발견되었습니다.
해결: Smart Path Clipping 로직을 적용하여, 짧은 경로에서는 접두사를 자동으로 제거하고 긴 경로에서만 활용하도록 지능형 경로 변환 기능을 탑재했습니다.

4. 레거시 바이너리 파손 방지 (Strict Exclusion Protocol)

현상: 구버전 바이너리(.xls, .ppt)는 내부 구조가 ZIP 패키지와 유사하여 최적화 엔진이 구조를 파괴하는 리스크가 있었습니다.
해결: Strict Exclusion 정책을 도입하여, 레거시 포맷은 패키지 최적화 대상에서 영구 제외하고 '재생형 저장(Regenerative Save)' 방식으로만 정제하도록 안전장치를 강화했습니다.

5. 금융권 수준의 원자적 파일 교체 (Atomic Replace)

현상: 파일 교체 중 시스템이 중단될 경우 원본 데이터가 손상될 위험이 존재했습니다.
해결: 백업(.bak) → 원자적 이동 → 무결성 검증 → 최종 삭제의 5단계 트랜잭션을 적용하여, 어떤 상황에서도 데이터 유실 없이 100% 자동 롤백되는 안정성을 확보했습니다.

📂 통합 병합-최적화 자동 파이프라인(Merge-Optimize Pipeline) 가이드 [v35.4.3]

문서를 하나로 합치는 과정에서 발생하는 용량 비대화 리스크를 근본적으로 해결하고, 필터 설정을 고도화한 병합-최적화 자동 연동 파이프라인 가이드입니다.

1. 통합 병합 시 '최적화' 연동 심층 분석

현상 분석: 기존 v35.4.2까지의 로직에서는 '통합 병합' 시 소스 파일들을 결합하여 새로운 문서를 생성하는 데 집중했습니다. 이 과정에서 개별 파일들이 가졌던 고화질 이미지나 메타데이터가 그대로 누적되어 병합된 파일의 용량이 비대해지는 현상이 발생했습니다.
조치 내용: run_merging 실행 시 병합 결과물이 생성되는 즉시 '패키지 최적화(_optimize_pkg)' 및 '심층 정제(_deep_clean)' 엔진이 자동 가동되도록 프로세스를 통합하였습니다.
대응 결과: 사용자가 별도로 최적화를 수행하지 않아도, 통합 병합 시 생성되는 최종 결과물은 이미지 압축(70% 품질), XML 경량화, 메타데이터 제거가 모두 완료된 최적 상태의 단일 파일로 출력됩니다.

2. 필터 확장자 개별 선택 및 누적 자동 입력 구현 (UI)

누적형 개별 선택: 기존의 묶음 방식 예시를 폐기하고, 사용자가 원하는 확장자만 정확히 골라 담을 수 있는 인터페이스를 탑재했습니다.
확장자 그룹 세분화: 오피스 3대 패밀리 뿐만 아니라 시스템 임시 파일(.bak, .tmp, .temp) 전체를 카테고리별로 노출합니다.

  • PPT: .ppt, .pptx, .pptm, .pps, .ppsx
  • Excel: .xls, .xlsx, .xlsm, .xlsb, .xltx, .xltm
  • Word: .doc, .docx, .docm
  • Temp: .bak, .tmp, .temp
지능형 누적 로직: 각 확장자를 클릭할 때마다 입력창 뒷부분에 자동으로 추가됩니다. 중복 입력 방지 로직이 적용되어 이미 목록에 있는 확장자는 추가되지 않으며, 쉼표(,)와 공백이 자동으로 정돈되어 입력됩니다.

💡 안정성: 병합 및 최적화 동시 수행 시의 리소스 해제 무결성을 100% 보증합니다.
🛡️ 초격차 경로 안전성(Path Safety) 및 원자적 교체(Atomic Replace) 가이드 [v35.4.2]

Windows의 물리적 한계인 MAX_PATH(260자) 제한을 완벽하게 극복하고, 파일 교체 시의 데이터 유실을 0%로 차단한 초정밀 하드닝 가이드입니다.

1. 경로 파괴 방지 (Path Safety Hardening): \\?\ 접두사 및 자동 단축

현상: 폴더 구조가 깊거나 파일명이 매우 길 때(240자 초과), 윈도우 표준 API와 오피스 COM은 파일을 읽지 못하고 오류를 뿜으며 중단됩니다.
해결: \\?\ 접두사(Long Path Support)를 UNC/로컬 경로에 지능적으로 적용하고, 전체 경로가 한계에 도달하면 확장자를 보존하며 파일명을 자동 단축(Truncation)하는 알고리즘을 탑재하여 어떤 깊은 구조에서도 '파일 없음' 오류 없이 완벽하게 작동합니다.

2. 원자적 확정 정리 (Atomic Replace): Backup-First 프로세스

현상: 최적화 완료 후 원본 파일을 교체하는 도중에 전원이 꺼지거나 네트워크가 끊기면 원본 파일이 손상(0KB)되는 리스크가 존재했습니다.
해결: 5단계 원자적 교체(Atomic Move) 프로세스를 도입했습니다.
① 사전 파일 잠금 확인 → ② .bak 임시 백업 생성 → ③ 원자적 파일 이동(Move) → ④ 무결성 최종 검증 → ⑤ 성공 시 백업 삭제 (실패 시 즉시 자동 롤백). 이를 통해 '확정 정리' 시 단 1바이트의 데이터 유실도 허용하지 않는 금융권 수준의 안전성을 확보했습니다.

🛡️ 강력한 UAC 통제 및 백그라운드 무응답(Hang) 교착 완벽 해결 가이드 [v2.9.27]

윈도우의 강력한 사용자 계정 컨트롤(UAC)과 UIPI 격리, 그리고 백그라운드(창 없음) 실행 중 발생하는 치명적 교착 상태(Deadlock)를 원천 극복한 아키텍처 가이드입니다.

1. 증상 및 원인 분석: 권한 충돌(-2147024156) 및 무한 대기(Hang)

권한 충돌: 대시보드가 관리자 권한으로 실행될 때, 파이썬이 DispatchEx로 엑셀/PPT를 제어하려 하면 윈도우 UIPI 보안 규칙에 의해 완전 차단되어 `-2147024156` (권한 접근 거부) 오류가 발생합니다.
무응답 교착 (Hang): 해결책으로 백그라운드 프로세스(CREATE_NO_WINDOW)를 띄웠으나, 파일 경로 증명이나 오피스 제한된 보기(Protected View) 경고 창이 백그라운드 내부에서 몰래 뜨게 될 경우, 사용자는 창을 볼 수 없어 '확인'을 누를 수 없고 스크립트는 영원히 무한 대기에 빠져 변환이 멈추거나 '건너뜀' 처리되는 치명적인 문제가 발견되었습니다.

2. 해결 기술: 윈도우 탐색기 비동기 위임 (ShellExecute + GetObject)

이 두 가지 문제를 동시에 해결하는 마스터키가 적용되었습니다.
1. 파이썬이 직접 프로그램을 호출하지 않고, win32api.ShellExecute(0, 'open', in_path, None, None, 0)를 통해 운영체제 쉘(탐색기)에게 파일을 백그라운드에서 열도록 완전히 비동기 위임합니다. 이는 UAC 격리나 백그라운드 감옥의 영향을 받지 않는 정상적인 열기 동작입니다.
2. 파일이 열리면 즉시 win32com.client.GetObject()를 호출하여(Lock-Free 모니커 결합), 활성화된 애플리케이션 객체를 가로채 변환을 진행합니다.

3. 다른 PC에서 동일 문제 발생 시 대응 및 원복 계획

[대응 방법] 다른 PC에서 오피스 버전이나 레지스트리 설정 차이로 이 위임 기술이 동작하지 않을 경우, 오피스 파일 자체의 '신뢰할 수 있는 위치' 옵션을 조정하거나, 대시보드를 일반 사용자 권한으로 실행할 것을 권장합니다.
[원복 계획(Rollback)] 만약 쉘 위임 기술이 심각한 호환성 문제를 일으킨다면, pattern_document_merger.py 및 group_cross_merger.py 코드 내의 app = get_app(...) 부분으로 즉각 폴백(Fallback)하도록 이중 안전장치가 설계되어 있습니다. 코드를 이전 버전 방식인 EnsureDispatch 위주로 되돌리면 즉시 원복됩니다.

🛠️ 하이브리드 자가 치유(Self-Healing) 및 시스템 자동 복구 가이드 [v3.5.7]

로컬 자산 유실이나 환경 훼손 시에도 별도의 수동 조작 없이 시스템이 스스로를 복구하는 2단계 하이브리드 엔진이 탑재되었습니다.

1. 지능형 자산 탐지 (Priority 0: Local Assets)

실행 시 상위 경로(../automated_app)와 현재 경로를 동시에 탐색하여, 인터넷이 연결되지 않은 보안 환경(절대 오프라인)에서도 내장된 인스톨러를 통해 파이썬 엔진과 라이브러리를 0순위로 복구합니다.

2. 클라우드 폴백 (Priority 1: Cloud Fallback)

만약 모든 로컬 자산이 삭제되거나 훼손된 경우, 시스템은 자동으로 클라우드 모드로 전환됩니다. PowerShell 엔진을 연동하여 공식 Python 3.13 인스톨러를 실시간 다운로드하고, PyPI 서버를 통해 누락된 핵심 라이브러리(Streamlit, pywin32 등)를 전수 재설치합니다.

3. 런처 구문(ParserError) 트러블슈팅

일부 시스템에서 발생하는 Unexpected token 오류를 해결하기 위해 런처 구조가 평탄화(Flat Label Sequence) 되었습니다. 만약 터미널에서 실행 시 숫자로 시작하는 파일명 때문에 오류가 난다면, 탐색기에서 직접 더블 클릭하거나 & ".\000..." 형식의 호출을 권장합니다.

🛡️ 고난도 보안 환경(UAC/Store Python) 엔진 연결 기술 가이드 [v34.1.14]

윈도우 스토어 버전 파이썬 및 UAC(사용자 계정 컨트롤) 환경에서도 오피스 자동화가 중단 없이 작동하도록 설계된 최신 기술(Deep Registry Hook)에 대한 상세 설명입니다.

1. 원인 분석: 권한 오류(0x800702E4)의 근본 원인

Microsoft Store에서 설치된 Python은 시스템과 격리된 AppContainer(Sandbox)라는 보안 영역에서 실행됩니다. 이로 인해 외부 프로세스인 PowerPoint/Excel 엔진을 OS 기본 이름(ProgID)으로 부를 때 윈도우 보안 정책이 이를 샌드박스를 넘으려는 위험 시도로 간주하여 차단합니다.

2. 해결 기술: CLSID Direct Hook & Force Bind

OS 이름 검색 과정을 우회하기 위해 엔진의 고유 주민번호와 같은 CLSID(범용 고유 식별자: {91493441...})를 사용하여 직접 통로를 개척합니다. 또한 EnsureDispatch 기술을 통해 레지스트리 바인딩 통로를 강제로 재등록하여 연결을 성공시킵니다.

3. Deep Polling: 타이밍 오류(0x800401E3) 완벽 제거

엔진이 가동되는 찰나의 타이밍을 놓치지 않도록 10초간 매초 정밀 추적(Deep Polling)을 수행합니다. 이를 통해 '작업을 사용할 수 없습니다'와 같은 초기화 에러 없이 어떠한 환경에서도 엔진 제어권을 안전하게 획득합니다.

4. 오피스 제품별 CLSID 매핑 정보
